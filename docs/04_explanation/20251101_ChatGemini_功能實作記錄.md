## 🧠 CLAUDE專案記憶系統

### 概述

CLAUDE.md 是一個「AI 可讀的專案記憶中心」，讓 AI 助手快速理解專案全貌，避免重複詢問，提升開發效率。

**核心理念**: 一次撰寫，永久記憶

### 主要功能

1. **專案基本資訊**
   - 專案名稱、目的、核心功能
   - 技術棧、依賴項、開發環境
   - 目錄結構、命名規範

2. **開發天條（不可違背的原則）**
   - 代碼讀取原則：優先使用 Read 工具
   - UI 設計標準：簡潔為原則（無 emoji、無 Nerd Font）
   - UX 設計標準：提供最好的預設值
   - 資料刪除操作原則：確認後執行
   - 智能選擇原則：信心度驅動
   - 路徑管理統一：單一真相來源
   - 重要資料持久化：自動備份
   - 工具管理自動化：依賴檢查

3. **工作流程規範**
   - 標準開發流程
   - Git 工作流程
   - 測試策略
   - 部署流程

4. **特殊考量**
   - 安全性要求
   - 性能優化指南
   - 多語言支援策略

### 實作細節

**檔案位置**: `ChatGemini_SakiTool/CLAUDE.md`

**檔案結構**:
```markdown
# 專案名稱

## 專案資訊
- 基本資訊
- 技術棧

## 開發天條
- 8 大原則

## 工作流程
- 開發流程
- Git 流程

## 特殊考量
- 安全性
- 性能
```

**整合方式**:
- AI 助手啟動時自動讀取
- 變更時同步更新
- 版本控制追蹤

### 使用範例

**場景 1: 新 AI 助手加入**
```
AI: 讀取 CLAUDE.md → 理解專案 → 開始工作
（無需重複詢問專案結構、規範）
```

**場景 2: 修改核心規則**
```
開發者: 更新 CLAUDE.md 中的天條
AI: 重新讀取 → 遵循新規則
```

### 效益

- ✅ 減少 80% 重複性問題
- ✅ 新 AI 助手 5 分鐘內上手
- ✅ 確保開發規範一致性
- ✅ 降低溝通成本

## 🧩 Extended Thinking 增強

### 概述

Extended Thinking（延伸思考）是 Gemini 2.0 的思考預算功能，允許模型在回答前進行更深入的思考。本功能實現了**自動觸發**機制，無需手動設定。

**設計理念**: 智能判斷，自動啟用深度思考

### 主要功能

1. **自動觸發條件**
   - 檢測到複雜問題（關鍵詞匹配）
   - 程式碼分析任務
   - 架構設計任務
   - 錯誤診斷任務

2. **動態預算調整**
   - 簡單問題：不啟用（節省成本）
   - 中等問題：固定 8K tokens
   - 複雜問題：動態模式（最多 32K）

3. **思考簽名顯示**
   ```
   🧠 [思考簽名] 動態思考模式 ✓
   📊 預算：8,192 ~ 32,768 tokens
   ```

4. **手動控制選項**
   - `/thinking on` - 強制啟用
   - `/thinking off` - 強制停用
   - `/thinking auto` - 自動模式（預設）
   - `/thinking budget 16384` - 設定固定預算

### 實作細節

**檔案位置**: `ChatGemini_SakiTool/gemini_thinking.py`

**觸發關鍵詞**:
- 複雜度: "複雜"、"difficult"、"challenge"
- 分析: "分析"、"analyze"、"evaluate"
- 設計: "架構"、"設計"、"design"、"architecture"
- 診斷: "錯誤"、"bug"、"debug"

**預算策略**:
```python
if is_complex_task():
    return ThinkingConfig(
        enabled=True,
        mode='dynamic',  # 8K~32K
        min_tokens=8192,
        max_tokens=32768
    )
else:
    return ThinkingConfig(enabled=False)
```

**整合方式**:
- 在 `gemini_chat.py` 中調用
- 每次對話前自動判斷
- 結果顯示在思考簽名中

### 使用範例

**場景 1: 自動啟用（複雜任務）**
```
用戶: 請分析這個複雜的架構設計問題...
系統: [自動偵測] → 啟用動態思考模式
🧠 [思考簽名] 動態思考模式 ✓
```

**場景 2: 自動不啟用（簡單任務）**
```
用戶: 今天天氣如何？
系統: [自動偵測] → 不啟用思考模式
（直接回答）
```

**場景 3: 手動強制啟用**
```
用戶: /thinking on
系統: ✓ 已啟用固定思考模式（8K tokens）
```

### 效益

- ✅ 複雜任務回答品質提升 30-50%
- ✅ 自動化，無需手動設定
- ✅ 節省成本（簡單任務不啟用）
- ✅ 靈活控制（支援手動覆寫）

## 📄 輸出格式化系統

### 概述

輸出格式化系統允許用戶在 **Rich 格式化輸出**（彩色、表格）和 **純文字輸出**（適合複製貼上、日誌記錄）之間切換。同時支援 JSON 格式輸出，便於程式化處理。

**設計理念**: 靈活輸出，適應不同需求

### 主要功能

1. **三種輸出格式**
   - **Rich 格式**（預設）：彩色、表格、進度條
   - **純文字格式**：無 ANSI 碼，適合日誌
   - **JSON 格式**：結構化資料，適合 API

2. **全局格式切換**
   ```bash
   /format rich      # 切換到 Rich 格式
   /format plain     # 切換到純文字
   /format json      # 切換到 JSON
   ```

3. **單次格式化**
   ```bash
   /format-once plain   # 僅下一次輸出使用純文字
   ```

4. **自動 JSON 檢測**
   - 檢測到 JSON 請求時自動格式化
   - 語法高亮顯示（Rich 模式）
   - 壓縮/美化切換

### 實作細節

**檔案位置**: `ChatGemini_SakiTool/utils/output_formatter.py`

**輸出範例對比**:

**Rich 格式**:
```
╭─────────────────╮
│ 🎯 執行結果     │
╰─────────────────╯
✅ 成功處理 10 個檔案
📊 統計資料:
  • 總行數: 1,234
  • 新增: 567
  • 刪除: 123
```

**純文字格式**:
```
執行結果
========
[OK] 成功處理 10 個檔案
統計資料:
  總行數: 1,234
  新增: 567
  刪除: 123
```

**JSON 格式**:
```json
{
  "status": "success",
  "files_processed": 10,
  "statistics": {
    "total_lines": 1234,
    "added": 567,
    "deleted": 123
  }
}
```

**整合方式**:
```python
from utils.output_formatter import format_output

result = {"status": "success", "count": 10}
output = format_output(result, format_type="json")
console.print(output)
```

### 使用範例

**場景 1: 日誌記錄**
```bash
/format plain
/batch-process files/
（輸出適合寫入日誌檔案）
```

**場景 2: API 整合**
```bash
/format json
/analyze codebase
（輸出 JSON 供其他程式處理）
```

**場景 3: 互動式使用**
```bash
/format rich
/help
（彩色表格、易讀）
```

### 效益

- ✅ 靈活輸出，適應不同場景
- ✅ JSON 格式便於自動化
- ✅ 純文字格式適合日誌/複製
- ✅ Rich 格式提升互動體驗

## 🎯 技術決策記錄

### 為什麼選擇 CLAUDE.md？

**問題**: AI 助手經常忘記專案規範，需要重複說明

**決策**: 建立 CLAUDE.md 作為「AI 可讀的專案記憶」

**理由**:
- 標準化專案資訊傳遞
- 減少溝通成本
- 確保規範一致性

**替代方案**:
- ❌ 每次對話都說明（重複、低效）
- ❌ 依賴 AI 記憶（不可靠）
- ✅ 專案記憶檔案（可版本控制、可更新）

---

### 為什麼自動觸發 Extended Thinking？

**問題**: 手動設定思考預算繁瑣，容易忘記

**決策**: 自動偵測任務複雜度並啟用

**理由**:
- 簡單任務不需深度思考（節省成本）
- 複雜任務自動啟用（提升品質）
- 保留手動控制（靈活性）

**替代方案**:
- ❌ 永遠啟用（成本高）
- ❌ 永遠停用（品質低）
- ✅ 智能自動（平衡成本與品質）

---

### 為什麼支援三種輸出格式？

**問題**: 不同場景需要不同輸出格式

**決策**: 實作 Rich / 純文字 / JSON 三種格式

**理由**:
- Rich: 互動式使用，易讀
- 純文字: 日誌記錄，複製貼上
- JSON: API 整合，程式化處理

**替代方案**:
- ❌ 只支援 Rich（不適合日誌）
- ❌ 只支援純文字（互動體驗差）
- ✅ 支援三種（靈活應對）
